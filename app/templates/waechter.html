Eskalation

Regelt, wann gemeldet wird — nicht was geprüft wird. Ziel: keine Fehlalarme bei einem einzelnen Aussetzer, aber auch kein Vergessen, wenn ein Problem tagelang steht.

1–10. Schützt vor Fehlalarm durch einen einzelnen Aussetzer (Neustart, kurze Netz-Störung). Preis: Der Alarm kommt später — bei {{ w.fail_threshold }} Läufen und {{ w.interval_min }} min Intervall frühestens nach {{ (w.fail_threshold - 1) * w.interval_min }} min. Wer sofort wissen will, dass die Instanz weg ist, trägt 1 ein.

1–10. Höher als 1 verhindert Alarm-Entwarnung-Alarm-Pingpong bei einer wackeligen Instanz.

0 = keine Erinnerung (wie bisher: einmal melden, dann Ruhe). Sonst wird erinnert, solange das Problem besteht — hilft, wenn die erste Meldung nachts unterging.

Empfohlen. Bei {{ w.repeat_min or 60 }} min heißt das: nach {{ w.repeat_min or 60 }} min, dann {{ (w.repeat_min or 60) * 2 }} min, dann {{ (w.repeat_min or 60) * 4 }} min … gedeckelt auf 24 h. Ohne Haken bleibt der Abstand konstant.

0 = nie anheben. Sonst geht die Meldung ab dieser Erinnerung eine Stufe lauter raus (z. B. „1 Hoch" → „2 Notfall" — Pushover wiederholt dann, bis du quittierst). Die Grundpriorität je Ereignis steht bei den Benachrichtigungen.

Meldet „behoben" samt Ausfalldauer, eine Stufe leiser als der Alarm. Ohne Haken landet die Entwarnung nur im Protokoll und im Webhook.

Webhook / n8n

Schickt ein JSON per POST an die URL (z. B. an einen n8n-Webhook-Node) — für eigene Automationen. Was rausgeht, wählst du unten aus.
{% for k, label in wh_kinds %} {% endfor %}

Alarm und Entwarnung kommen einmal je Vorfall — der Flapping-Schutz aus der Eskalation gilt auch hier, der Alarm kommt also erst nach {{ w.fail_threshold }} schlechten Läufen. Erinnerungen feuern nur, wenn oben ein Erinnerungsabstand gesetzt ist{% if not w.repeat_min %} — steht gerade auf 0, es käme also keine{% endif %}. Digest und Report gehen auch dann über den Webhook, wenn ein Profil gar keinen Benachrichtigungs-Kanal hat — n8n allein genügt. Der Testknopf feuert immer, unabhängig von dieser Auswahl.

Payload — bei jeder Art identisch aufgebaut:

{
  "source":         "paperless-generator-portal",
  "portal_version": "{{ portal_version }}",
  "kind":           "alarm",       // alarm | reminder | recovery | digest | report | test
  "event":          "downtime",    // downtime | drift | task_fail | asn_gap | duplicate
                                   //   bzw. digest | report | test
  "profile":        "Meine Instanz",
  "status":         "bad",         // bad | ok | info | test
  "detail":         "Paperless nicht erreichbar unter http://…",
  "ts":             "2026-07-15T14:03:11"
}

kind sagt was passiert ist, event welcher Check es betrifft. Beides ist nötig: Alarm und Erinnerung haben denselben status: "bad" — ohne kind könntest du sie in n8n nicht auseinanderhalten. Verzweige also auf {{ '{{' }} $json.kind {{ '}}' }}.

⚠️ n8n-Stolperfallen: Der Webhook-Node muss auf HTTP Method = POST stehen — bei GET antwortet n8n mit 404 „This webhook is not registered for POST requests". Und die Test-URL lauscht nur, solange der Workflow im Editor offen ist und du „Listen for Test Event" geklickt hast: für den Dauerbetrieb den Workflow aktivieren und die Production-URL (/webhook/… statt /webhook-test/…) eintragen.

Prometheus /metrics

Gibt den Wächter-Stand als Prometheus-Exposition aus — für Grafana, Alertmanager & Co. Rein lesend und ohne eigene Abfrage an deine Instanzen: der Endpunkt gibt nur wieder, was der letzte Prüflauf ohnehin erhoben hat. Ohne Haken antwortet /metrics mit 404.
{% if mc.enabled and mc.token %}

/metrics hängt nicht am Portal-Login — Prometheus hat keine Sitzung und würde auf /login umgeleitet. Stattdessen schützt dieses Token den Endpunkt: ohne passenden Authorization: Bearer-Header gibt es 401. Eingeloggt im Browser kommst du auch ohne Token dran. Das Token steht im Klartext in config.json und geht bei http:// unverschlüsselt über dein Netz — im LAN vertretbar, über das Internet nur hinter einem TLS-Reverse-Proxy.

Scrape-Konfiguration (prometheus.yml):

scrape_configs:
  - job_name: paperless-portal
    scrape_interval: 60s
    metrics_path: /metrics
    authorization:
      credentials: {{ mc.token }}
    static_configs:
      - targets: ['{{ request.host }}']

60 s reichen. Häufiger bringt nichts: die Werte ändern sich nur im Wächter-Takt ({{ w.interval_min }} min), die Dokument-Zähler sogar nur ~1×/Stunde. Jedes Sample trägt seinen Erhebungs-Zeitstempel mit (…_timestamp_seconds).
